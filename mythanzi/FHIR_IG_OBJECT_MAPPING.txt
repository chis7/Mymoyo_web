Recommended IG artifacts
- StructureDefinition/mythanzi-resource
  Base shared profile applied to every exported MyThanzi FHIR resource through meta.profile.
- StructureDefinition/mythanzi-province-location
  Profile on Location for province records.
- StructureDefinition/mythanzi-district-location
  Profile on Location for district records.
- StructureDefinition/mythanzi-facility-location
  Profile on Location for service delivery facilities.
- StructureDefinition/mythanzi-healthcare-service
  Profile on HealthcareService for the service catalogue.
- StructureDefinition/mythanzi-person-identity
  Profile on Person for one real person who may have multiple login personas.
- StructureDefinition/mythanzi-client-patient
  Profile on Patient for client UserProfile records.
- StructureDefinition/mythanzi-worker-practitioner
  Profile on Practitioner for non-client UserProfile records.
- StructureDefinition/mythanzi-appointment
  Profile on Appointment for booked, fulfilled, or missed visits.
- StructureDefinition/mythanzi-provenance
  Profile on Provenance for project audit log events.
- StructureDefinition/mythanzi-fhir-resource-version
  Internal persistence/audit object. Usually not exported as a clinical FHIR resource; document it as IG implementation infrastructure.
- StructureDefinition/person-identity
  Extension linking Patient or Practitioner back to a Person resource.
- StructureDefinition/worker-facility
  Extension linking Practitioner records to their assigned facility Location.
- NamingSystem/facility-code
  Identifier system for locations.Facility.code.
- NamingSystem/service-code
  Identifier system for locations.Service.code.
- NamingSystem/client-reference
  Identifier system for users.UserProfile.reference_number.
- CodeSystem/mythanzi-user-role
  Codes: client, mobiliser, chw, provider, supervisor, admin.
- ValueSet/mythanzi-user-role
  Permitted UserProfile.role codes.
- CodeSystem/mythanzi-appointment-purpose
  Codes: clinical_review, lab_collection, medication_refill, follow_up.
- ValueSet/mythanzi-appointment-purpose
  Permitted Appointment.visit_purpose codes.
- CodeSystem/mythanzi-appointment-status
  Local status codes: upcoming, completed, missed. Map to FHIR Appointment.status booked, fulfilled, noshow.
- CodeSystem/mythanzi-submission-type
  Codes for self-risk assessment, self-test report, side-effect report, and clinic feedback.

Object mappings

1. locations.Province
- FHIR resource: Location
- Recommended profile: StructureDefinition/mythanzi-province-location
- Logical id pattern: province-{pk}
- Field mapping:
  - name -> Location.name
  - fixed status active -> Location.status
  - fixed mode instance -> Location.mode
  - type Jurisdiction / Province -> Location.type
- Relationships:
  - Parent of District resources.
- Current implementation: implemented in api.fhir._location_for_province.

2. locations.District
- FHIR resource: Location
- Recommended profile: StructureDefinition/mythanzi-district-location
- Logical id pattern: district-{pk}
- Field mapping:
  - name -> Location.name
  - fixed status active -> Location.status
  - fixed mode instance -> Location.mode
  - fixed type District -> Location.type.text
  - province -> Location.partOf reference to Location/province-{province.pk}
- Relationships:
  - Child of Province.
  - Parent of Facility resources.
- Current implementation: implemented in api.fhir._location_for_district.

3. locations.Service
- FHIR resource: HealthcareService
- Recommended profile: StructureDefinition/mythanzi-healthcare-service
- Logical id pattern: service-{pk}
- Field mapping:
  - is_active -> HealthcareService.active
  - name -> HealthcareService.name
  - code -> HealthcareService.identifier using NamingSystem/service-code
  - description -> HealthcareService.comment
- Relationships:
  - Many-to-many with Facility. Consider HealthcareService.location references if service availability at facilities must be exported directly.
- Current implementation: implemented in api.fhir._healthcare_service.

4. locations.Facility
- FHIR resource: Location
- Recommended profile: StructureDefinition/mythanzi-facility-location
- Logical id pattern: facility-{pk}
- Field mapping:
  - name -> Location.name
  - fixed status active -> Location.status
  - fixed mode instance -> Location.mode
  - level or "Facility" -> Location.type.text
  - code -> Location.identifier using NamingSystem/facility-code
  - latitude -> Location.position.latitude
  - longitude -> Location.position.longitude
  - district -> Location.partOf reference to Location/district-{district.pk}
- Relationships:
  - Child of District.
  - Linked from Practitioner through worker-facility extension.
  - Linked from Appointment.supportingInformation.
  - Linked from ClinicFeedbackSubmission if that object is exported.
- Current implementation: implemented in api.fhir._location_for_facility.

5. django.contrib.auth.models.User
- FHIR resource: not exported directly.
- Recommended IG treatment: implementation source object only.
- Field mapping through UserProfile:
  - first_name, last_name, username -> Patient.name or Practitioner.name
  - email -> Patient.telecom or Practitioner.telecom
  - is_active combines with UserProfile.is_active -> Patient.active or Practitioner.active
- Relationships:
  - One-to-one with UserProfile.
- Current implementation: User fields are embedded through UserProfile in api.fhir._person_resource.

7. users.UserProfile with role = client
- FHIR resource: Patient
- Recommended profile: StructureDefinition/mythanzi-client-patient
- Logical id pattern: user-{user_id}
- Field mapping:
  - reference_number -> Patient.identifier using NamingSystem/client-reference
  - user first_name, last_name, username -> Patient.name
  - phone -> Patient.telecom phone
  - user.email -> Patient.telecom email
  - date_of_birth -> Patient.birthDate
  - is_active and user.is_active -> Patient.active

  - PersonIdentity links to this Patient when present.
- Current implementation: implemented in api.fhir._person_resource.

8. users.UserProfile with role != client
- FHIR resource: Practitioner
- Recommended profile: StructureDefinition/mythanzi-worker-practitioner
- Logical id pattern: user-{user_id}
- Field mapping:
  - reference_number -> Practitioner.identifier using NamingSystem/client-reference
  - user first_name, last_name, username -> Practitioner.name
  - phone -> Practitioner.telecom phone
  - user.email -> Practitioner.telecom email
  - date_of_birth -> Practitioner.birthDate
  - is_active and user.is_active -> Practitioner.active
  - role display -> Practitioner.qualification.code.text
  - person_identity -> person-identity extension with reference to Person/person-{pk}
  - facility -> worker-facility extension with reference to Location/facility-{pk}
- Relationships:
  - Appointment.created_by can point to this Practitioner.
  - AuditLog.actor can point to this Practitioner.
- Current implementation: implemented in api.fhir._person_resource.
- IG note:
  - If staff roles need formal authorization, consider PractitionerRole in a future version. Current code uses Practitioner plus qualification and extension.

9. users.Appointment

11. api.FHIRResourceVersion

- Field mapping:
  - resource_type -> stored FHIR resource type
  - logical_id -> stored FHIR logical id
  - version_id -> stored FHIR version id

Potential future FHIR resources
- PractitionerRole
  Use when role, organization, facility, service, and availability need formal representation beyond Practitioner.qualification and worker-facility extension.
- Organization
  Use if facilities need ownership/administration separate from physical Location.
- Task
  Use for requested follow-up from side-effect reports or clinic feedback.
- ServiceRequest
  Use for clinical follow-up orders, referrals, or requested interventions.
- Questionnaire
  Use to publish formal definitions for self-risk assessment, self-test report, side-effect report, and clinic feedback forms.
- Observation
  Use for derived scores, test results, satisfaction ratings, or summarized form outcomes.
- RiskAssessment
  Use for HIV prevention risk level if a clinical decision summary is required.
- AdverseEvent
  Use for side-effect reporting tied to prevention medicines or products.

Implementation notes
- The active project code currently uses one broad profile URL, StructureDefinition/mythanzi-resource, for all exported resources.
- A fuller IG should replace or supplement that broad profile with model-specific profile URLs listed above.
- FHIR sync is resilient by design: api/signals.py records versions after transaction commit and suppresses sync failures so application saves continue.
- HAPI sync strips local meta.versionId and meta.lastUpdated before PUT because the HAPI server owns persisted resource version metadata.
- Current HAPI configuration is R5, and api/hapi.py includes an R5 repair for Appointment.serviceType.
